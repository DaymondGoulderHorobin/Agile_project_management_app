# Build Backlog

## Backlog rules

This is the ordered, documentation-only implementation backlog for the first working company version. Work is owned by exactly one vertical slice. A later slice may consume or extend an earlier capability, but it must not recreate it.

- Slice identifiers are `EPIC-S1` through `EPIC-S6` and correspond exactly to [08-implementation-roadmap.md](./08-implementation-roadmap.md).
- `US` denotes user value, `T` a technical enabler, `SEC` a security/privacy task, and `TEST` a verification task.
- Requirement IDs are defined in [01-product-requirements.md](./01-product-requirements.md); demo IDs are defined in [13-demo-journey.md](./13-demo-journey.md).
- “Mechanical Codex work” means bounded implementation that is suitable for Codex after the relevant human decision and execution-plan approval. It does not authorise execution during planning.
- No calendar or effort estimate is implied. Order is dependency order.

Assessment convention: each row’s “Complexity, uncertainty, and verification” cell identifies the item-specific logic/integrity or architectural risk and verification burden. If it does not name a separate human decision, the item inherits the explicit human gates stated beneath its epic; if it does not name an external integration, integration uncertainty is “none beyond the epic gates.” Thus every item has a declared logic/risk, human-decision, integration-uncertainty, verification, and mechanical-work treatment without pretending to have an effort estimate.

## Global definition of done

An item is done only when its acceptance criteria, authorisation rules, tenant-aware schema/API contracts, audit and outbox effects, failure handling, automated tests, accessibility needs, documentation, and observability are complete. External effects require durable intent, idempotency, and reconciliation. AI output requires persistent origin and proposal status. Evidence links must target immutable fragments or versions. Security and privacy tests are release criteria, not follow-up work.

### Reuse without duplicated ownership

- EPIC-S1 owns the generic invitation/token, audit/outbox, tenant and membership foundations; EPIC-S2 owns project-guest grants, next-action UX and discovery participation built on them.
- EPIC-S2 owns AI job/origin/provenance infrastructure and question suggestion; EPIC-S3 and EPIC-S4 own their distinct artifact and backlog proposal use cases/evaluations.
- EPIC-S3 owns the Project approval engine and immutable snapshot mechanics; EPIC-S4–EPIC-S6 only configure new subjects/policies and reuse the engine.
- EPIC-S5 owns execution reviews and cycle cancellation; EPIC-S6 consumes those records and invokes cancellation only through an approved change impact.
- EPIC-S6 alone owns change proposal/impact/application and release-record capabilities. No earlier “foundation” creates those domain records.
